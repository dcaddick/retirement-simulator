# Retirement Income Simulator v1.02 Interaction Polish Design

## Goal

Make live modelling calmer and more efficient by reducing chart-tooltip noise, compacting lump-sum controls, making table height adjustable, fixing the truncated left panel, and preserving the user's exact interface position during recalculation.

The dependency-free single-HTML-file architecture and current chart height remain unchanged.

## Concise chart tooltips

Tooltips contain two lines only.

Normal income segment:

```text
2033 · ages 70 and 68
$17,475 this year from Jane super drawdown, contributing to total $80,000
```

Lump-sum segment:

```text
2031 · ages 68 and 66
$50,000 for European holiday from John super + Jane super
```

The lump-sum funding text uses actual sources drawn, not only the requested source. Remove target, target-met, shortfall and other redundant lines from segment tooltips.

## Table-height control

Add a horizontal separator beneath the table. Pointer dragging and keyboard controls adjust only the table's visible height; chart height does not change. Double-click resets to the default. Persist the selected height locally.

The table keeps its horizontal and vertical scrolling. The divider expands the table downward into the available workspace rather than taking space from the chart.

## Viewport and left-panel scrolling

Use the page's natural four-row structure—top disclaimer, header, workspace and bottom disclaimer—as a viewport-height grid. The workspace and its children use `min-height: 0`, allowing the left controls and main stage to scroll internally without clipping their final content.

Mobile retains document-flow stacking rather than nested desktop scrollers.

## Compact lump-sum editor

Each lump-sum item uses a two-column, two-row field grid:

- Amount | Reason
- Intended Year | Fund from

A compact footer contains:

- `Include & calculate on chart` checkbox; and
- Remove button.

The item retains a clear boundary from adjacent withdrawals but avoids the current unused vertical space.

## Enable or disable modelling

Advance the deterministic scenario schema from 7 to 8. Add boolean `enabled` to every lump-sum withdrawal.

- New and fictional-sample withdrawals default to enabled.
- Schema 7 scenarios migrate missing values to enabled, preserving current results.
- Disabled withdrawals remain in local storage and JSON exports.
- Disabled withdrawals do not reduce assets, create Events, render chart blocks or affect compatibility checks.
- Re-enabling immediately restores their modelling effect.

Monte Carlo v0.5 continues to reject active lump sums explicitly. Disabled lump sums may be discarded by its adapter because they have no deterministic cash-flow effect.

## Preserve modelling context

Before any full control or dashboard rerender, capture and restore:

- Income/Balance tab;
- open/closed state of every left-panel section;
- left-panel scroll position;
- main-stage/page scroll position;
- table horizontal and vertical scroll positions;
- chart height and table height; and
- focused control, identified by `id` or `data-path`, plus its text-selection range when the same control still exists after rerender.

Pure value edits should use the smallest possible update path. Recalculation still happens live, but the interface must not jump to another section, table row or viewport position.

## Mercer-style survival ribbon

Refine the existing Person 1 ribbon on both charts:

- one pale, rounded, full plot-width band inside the chart;
- exact label `Chance of living to each age` at the left;
- milestone ticks rising from the band with 80%, 50%, 20% and 5% labels;
- conventional five-year age axis immediately beneath the band; and
- thresholds outside the financial model horizon remain hidden.

The financial projection horizon remains authoritative and is not extended for survival display alone.

## Verification

Extend regression coverage for:

- schema 7-to-8 migration;
- enabled and disabled lump sums;
- concise tooltip formatting;
- compact 2x2 editor structure;
- persistent accessible table-height divider;
- viewport-grid and scroll-preservation hooks;
- one Person 1 Mercer-style ribbon; and
- unchanged chart-height persistence.

Run deterministic and Monte Carlo suites, whitespace and file-size checks, then perform one real-browser pass in desktop light/dark and narrow layouts. Exercise a left-panel toggle while the table is scrolled both vertically and horizontally and confirm the modelling context stays fixed.

## iPad and iOS 26 compatibility

iPad Safari on iOS 26 is a release requirement. Implementation must:

- avoid hover-only access to information; chart inspection remains available by pointer/tap and keyboard where a keyboard is attached;
- use Pointer Events with `touch-action: none` on draggable dividers so finger dragging does not scroll the page accidentally;
- provide at least 44 CSS-pixel touch targets for primary buttons, checkboxes and splitter hit areas at iPad breakpoints;
- avoid browser APIs without the existing compatibility shim or a safe fallback;
- keep form controls at 16 CSS pixels or larger on touch layouts to prevent Safari input zoom;
- respect iPad safe-area insets for the top and bottom disclaimer bars;
- support portrait and landscape at representative 820x1180, 1024x1366, 1180x820 and 1366x1024 viewports; and
- retain horizontal table scrolling without trapping vertical page or panel scrolling.

Automated browser checks on Windows provide responsive and interaction evidence but cannot emulate WebKit on real iOS. Final platform acceptance is a manual pass on the user's iPad running iOS 26: load/reset sample, expand and collapse sections, toggle a lump sum, drag both horizontal dividers, horizontally and vertically scroll the table, inspect normal and lump-sum chart segments by tap, rotate portrait/landscape, reload, and confirm saved divider heights and scenario state return.

The v1.02 README screenshot must show the fictional sample in the application's default dark theme. Clear any locally persisted light-theme preference before capture and visually confirm the toggle offers `Light theme`, proving dark mode is active.

## Out of scope

- Automatic or background update checks. These require an external request and would weaken the project's no-background-callback promise.
- Extending the model horizon solely to display out-of-range survival milestones.
- Splitting the application into multiple runtime files or adding dependencies.
