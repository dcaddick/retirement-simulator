# Retirement Income Simulator v1.03 Collapsible Lump Sums Design

## Goal

Reduce the space used by completed lump-sum entries so the left panel remains practical during live modelling, while keeping each entry immediately available for editing or removal.

## Interaction

Each lump sum has two presentation states.

### Collapsed

A completed entry appears as one compact, aligned row:

```text
›   $100K   Holiday   Jun 2032   ☑
```

- The left chevron expands the entry.
- The amount is abbreviated for display only; the exact stored value is unchanged.
- The reason remains readable and truncates gracefully when space is constrained.
- The intended date is displayed as abbreviated month plus four-digit year, for example `Jun 2032`.
- The right checkbox controls whether the withdrawal is included in modelling.
- Unchecked entries remain visible but use the existing subdued disabled treatment.

The summary uses explicit grid columns rather than comma-separated text so multiple entries align neatly. The checkbox retains an accessible name equivalent to `Include <reason> in modelling`.

### Expanded

The chevron points down and reveals the existing v1.02 editor:

- Amount;
- Reason;
- Intended date;
- Fund from;
- modelling checkbox; and
- Remove button.

Editing continues to recalculate live using the existing context-preservation behaviour. Collapsing returns to the refreshed summary values.

## Default state

- Existing valid entries render collapsed when the scenario opens or is reset.
- A newly added incomplete entry renders expanded so it can be completed.
- Collapse state is interface-only and is not exported in scenario JSON.
- Rerenders preserve each entry's open or closed state by stable lump-sum ID.

## Intended month and schema

Advance the deterministic scenario schema from 8 to 9 and add integer `month` (`1` through `12`) to each lump-sum withdrawal. The editor replaces Intended Year with an Intended Date month/year control. New sample and user-created entries default to the current month when their intended year is the current year, otherwise January.

Schema 8 scenarios migrate existing year-only lump sums to January of their stored year. JSON import, export and local persistence retain the selected month.

The financial model remains annual: the month improves planning clarity and the collapsed summary but does not change the withdrawal's annual calculation timing. Monte Carlo's adapter accepts schema 9 under the same rule as schema 8: active lump sums remain unsupported and disabled lump sums may be discarded.

## Responsive and accessible behaviour

- Desktop and iPad layouts retain aligned columns for chevron, amount, reason, date and checkbox.
- On narrow phones, the reason may truncate before the amount or date; controls remain reachable without hover.
- Chevron and checkbox targets meet the existing iPad 44 CSS-pixel touch-target requirement.
- Chevron buttons expose `aria-expanded` and identify the entry they control.
- Keyboard focus remains visible and Space/Enter operate the chevron and checkbox normally.

## Verification

Extend automated coverage for:

- valid entries defaulting to collapsed;
- new incomplete entries defaulting to expanded;
- chevron expansion exposing edit and remove controls;
- summary amount abbreviation and `Mon YYYY` date formatting;
- schema 8-to-9 migration with January as the explicit legacy default;
- month validation and JSON round-tripping;
- checkbox toggling the existing `enabled` value without requiring expansion;
- disabled-row styling;
- open/closed state surviving recalculation by stable item ID; and
- no schema or calculation-result change from collapse state alone.

Perform one browser pass in desktop and representative iPad widths with several entries to confirm columns align, long reasons truncate cleanly, and checkbox/chevron touch targets do not overlap.

## Out of scope

- Changes to lump-sum funding, tax, drawdown or chart logic.
- Month-level cash-flow timing within the annual financial model.
- A general accordion redesign for other input sections.
