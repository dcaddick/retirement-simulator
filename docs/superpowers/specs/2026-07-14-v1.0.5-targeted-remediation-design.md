# v1.0.5 Targeted Remediation Design

**Status:** Approved for implementation planning on 2026-07-14

**Release identity:** The release is v1.0.5. The executable continues the existing compact on-screen convention and displays `v1.05`.

## Purpose

v1.0.5 is a targeted correction release. It adds a simple projection-table export, tidies the Returns & Inflation presentation, corrects three source-backed engine or validation issues, and makes the remaining modelling limitations visible for later review. It does not expand the simulator into a comprehensive portfolio, mortality, tax, or social-security model.

## Release Boundary

### Included

1. Export the currently rendered projection table as CSV.
2. Recompose the table toolbar so the dollar display selector sits in a compact right-aligned control group and the Export CSV button is the far-right action.
3. Make the Treasury real-return readouts compact and left aligned.
4. Validate five projection-critical numeric fields that are currently accepted without checks.
5. Correct the Age Pension couple income-test taper from 40 cents to 50 cents combined per dollar above the couple free area.
6. Restore bracket creep for the fixed nominal income-tax brackets and LITO while retaining the simulator's existing indexed approximation for Medicare and SAPTO.
7. Update release identity, archive, tests, README, methodology, and changelog.
8. Add a durable deferred-review register for every verification-pack item that is not remediated in v1.0.5.

### Deferred

The following are deliberately not implemented in v1.0.5:

- share-price growth, dividends, and franking credits;
- Age Pension eligibility for age-gapped couples;
- mortality-driven first-death and survivor-state transitions;
- taxation of super benefits before age 60;
- capital-loss offset and carry-forward;
- owner attribution for other taxable income;
- CGT-to-income-waterfall reconciliation or redesign; and
- other verification-pack observations not explicitly included above.

These items require a broader data-model change, a professional or owner policy decision, an accounting trace, or a separately bounded design. They will be summarised in `docs/DEFERRED-REVIEW.md` with their finding reference, observed behaviour, likely impact and direction, reason for deferral, required reviewer or decision, and suggested next action. The public-facing summary must not reproduce confidential pack content beyond what is necessary to identify and manage the limitation.

## User Interface Design

### Projection table toolbar

The toolbar retains the dollar-basis note on the left. The recalculation stamp moves into the right-hand side next to a compact actions group. The actions group contains:

1. the existing Display label and selector; and
2. an `Export CSV` button at the far-right edge.

On narrower layouts the right-hand controls may wrap as a unit without changing their order. Focus styles and coarse-pointer minimum sizes continue to follow the existing interface rules.

### CSV export

The export is intentionally a screen export rather than a raw-data export. It reads the headings and cells from the rendered `#tbl` element, so the file contains exactly the columns and formatted values visible to the user, including:

- the current people's names;
- columns that appear only when optional income or assets exist;
- the active Today's dollars or Nominal future-dollars mode;
- formatted currencies, rates, events, and status markers; and
- the currently projected year range.

The exporter converts the table into a matrix, serialises it as RFC-style CSV, and downloads it as UTF-8 with a byte-order mark for Excel compatibility. Fields containing commas, quotation marks, or line breaks are quoted and embedded quotes are doubled. Cells beginning with spreadsheet formula trigger characters (`=`, `+`, `-`, or `@`) are prefixed safely before serialisation. This is the one deliberate exception to byte-for-byte visible-text fidelity.

The filename uses a filesystem-safe scenario name, the active basis (`todays-dollars` or `future-dollars`), and the export date. No spreadsheet or third-party runtime is added.

If no valid rendered projection exists, the export does not create a file and reports the issue through the existing scenario-message pattern. A successful export produces a short success message.

### Returns & Inflation presentation

Each accumulation and retirement return keeps its current editable nominal percentage field. The calculated readout beneath it becomes a compact, left-aligned two-line block:

- `Real return: +3.5% long-term`
- `2026: +1.0%` followed by the existing `near-term inflation` badge when the Treasury near-term rate differs from the long-run rate.

Manual inflation continues to show only the applicable real-return result. The readout remains live as the return or inflation assumptions change. The change is typographic and structural only; it does not alter return calculations.

## Engine Corrections

### Projection-critical validation

`validateScenario` adds finite-number and range checks matching the current UI constraints:

| Field | Rule |
|---|---|
| `people.*.salary` | finite and `>= 0` |
| `people.*.sgPct` | finite and `0..30` |
| `people.*.accumulationReturnPct` | finite and `-20..30` |
| `people.*.retirementReturnPct` | finite and `-20..30` |
| `assumptions.manualInflationPct` | finite and `0..20` when inflation mode is `manual` |

Treasury mode does not fail solely because an unused manual-inflation value is absent. Invalid imported values are rejected before projection rather than defaulted or coerced, avoiding both NaN cascades and plausible-looking silent zeroes. The schema remains version 10 because no persisted field is added or reinterpreted.

### Age Pension income taper

`SERVICES_AUSTRALIA_2026.incomeTaperCouple` changes from `0.40` to `0.50`. The couple free area and other pension constants are unchanged. Tests cover values below, at, and above the free area and verify that the assets test remains independent.

The correction is supported by Services Australia's description that each dollar above the couple income free area reduces each partner's payment by 25 cents, or 50 cents combined.

### Bracket creep

The tax schedules remain fixed in nominal dollars after the legislated 2027 rate change. Projection tax is separated by basis without rewriting the low-level boundary functions:

- income-tax brackets run against nominal taxable income;
- LITO runs against nominal taxable income;
- Medicare continues to run against real taxable income and the existing real thresholds;
- SAPTO continues to run against real rebate income and the existing real thresholds; and
- the nominal income-tax and LITO amounts are divided by the year's inflation factor before they are combined with the real Medicare and SAPTO amounts.

This preserves the current calculation structure while making a constant real income move through fixed nominal brackets over time. The existing `netTax` boundary behaviour remains available for direct tests; a projection-specific helper expresses the mixed-basis calculation explicitly and is also used for CGT incremental-tax comparisons.

The methodology will state the assumption: no unlegislated post-2027 bracket changes are forecast, while Medicare and SAPTO thresholds retain the simulator's existing indexed approximation. This is a deterministic policy baseline, not a prediction.

## Deferred Review Register

`docs/DEFERRED-REVIEW.md` is a living register rather than a release-note dump. Each item includes:

- identifier or verification-pack reference;
- current simulator behaviour;
- why it matters and the likely direction of bias, where known;
- why it is outside v1.0.5;
- the professional review, owner decision, trace, or design work required;
- a concrete next action; and
- status, initially `Deferred`.

README and CHANGELOG link to this register. Completing a future item updates its status and points to the release or decision that resolved it; the original entry is not deleted.

## Release and Compatibility

- Update title, visible version, storage key, and explanatory copy to v1.05.
- Retain older storage keys so existing local scenarios are discovered.
- Do not change schema version 10.
- Archive the exact pre-change v1.04 executable as `archive/retirement-simulator-v1.0.4.html` before modifying the main executable.
- Update the archive index, README, methodology, changelog, and relevant screenshots or version references.
- Preserve the single-file, offline-first architecture and current Content Security Policy.

## Error Handling and Security

- Validation failures prevent projection and export through the existing error path.
- CSV generation operates only on text already rendered by the application.
- CSV cells are protected against formula injection even when a scenario-controlled name or event starts with a spreadsheet trigger character.
- Object URLs created for download are revoked after use.
- No external network request or library is introduced for export.
- Existing HTML escaping remains required for all rendered scenario-controlled text.

## Verification Strategy

### Automated tests

Add or update deterministic tests for:

1. v1.05 title, visible version, storage-key compatibility, archive presence, and documentation references;
2. CSV quoting, embedded quotes, newlines, UTF-8 content, and formula-injection protection;
3. table-matrix fidelity for the active visible headings and cells;
4. export refusal when no valid table exists;
5. missing, non-numeric, non-finite, below-minimum, above-maximum, and valid boundary values for all five newly validated fields;
6. Treasury-mode handling of an unused manual-inflation value;
7. Age Pension income-test results below, at, and above the couple free area;
8. increasing real tax over time for constant real income under frozen nominal brackets;
9. the 2026 and 2027 legislated tax-rate transition;
10. CGT incremental-tax calculation under the mixed-basis projection helper; and
11. all existing migrations, tax boundaries, drawdown rules, invariants, and regression cases.

### Visual verification

After automated tests pass, perform one browser-based visual QA pass of the running local file:

- dark theme at a normal desktop width;
- light theme at a normal desktop width;
- a narrow responsive width;
- Treasury and manual inflation modes;
- the right-aligned Display and Export CSV controls; and
- the left-aligned return readouts for both people.

Fix only glaring layout, readability, wrapping, focus, or contrast defects found in that pass. Capture the final screenshot for handoff.

## Acceptance Criteria

v1.0.5 is complete when:

- the user can export the visible table to a safe, Excel-compatible CSV;
- the toolbar ordering and return readouts match the approved layout;
- malformed critical inputs cannot enter projection math;
- the pension taper and bracket-creep corrections are covered by focused tests;
- deferred verification items are discoverable and actionable in their own register;
- the executable remains a standalone offline HTML file;
- the full deterministic and Monte Carlo test suites pass; and
- final browser QA shows no glaring regression in dark, light, or narrow layouts.
